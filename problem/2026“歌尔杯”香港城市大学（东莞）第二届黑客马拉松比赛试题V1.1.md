# 2026"歌尔杯"香港城市大学（东莞）第二届黑客马拉松比赛试题

## 赛题：基于AI Agent的IT功能全链路自动化开发系统

---

### 一、赛题背景

在当今软件开发中，从产品需求到代码实现的流程**至少涉及以下环节**：需求分析、概要设计、编码实现、单元测试等。传统模式下，每个环节依赖人工完成，效率低、周期长、易出错。

随着大语言模型（LLM）技术的成熟，AI Agent（智能体）已具备代码生成、文档编写、设计规划等能力。本赛题要求参赛队伍基于 **多智能体协同机制**，设计并实现一套《基于AI Agent的IT功能全链路自动化开发系统》，实现从“产品规格说明书”到代码生成的全自动化流程。

---

### 二、赛题目标

构建一个《基于AI Agent的IT功能全链路自动化开发系统》（使用此系统，通过“产品规格说明书”自动开发“应用系统”），该系统能够：

1. **输入**：产品规格说明书（Markdown格式）
2. **处理**：通过多个专用Agent协作，依次完成“概要设计、代码生成、单元测试”环节，每个环节是一个专用Agent
3. **输出**：可运行的源代码、代码包括至少以下3个Agent：概要设计、代码生成、单元测试
4. **验证**：使用《基于AI Agent的IT功能全链路自动化开发系统》，通过一个“产品规格说明书”测试用例，生成此“产品规格说明书”对应的应用系统（生成的代码不要求达到100%可运行，可以由人工调整至可运行状态）

---

### 三、核心功能要求

参赛队伍需实现以下核心功能模块：

#### 模块1：流程编排引擎（必做）

实现一个流程编排系统，能够管理多个执行节点的生命周期，包括：

- **节点定义与管理**：定义每个节点的输入、输出、依赖关系
- **状态追踪**：追踪每个节点的执行状态（等待中、执行中、已完成、失败）
- **依赖管理**：确保上游节点完成后才启动下游节点
- **执行模式支持**：至少支持以下两种模式之一
  - **自动模式**：按依赖顺序自动执行所有节点  
  - **手动模式**：每个节点手动确认后再执行


#### 模块2：概要设计Agent（必做）

基于“产品规格说明书”（**输入参数**），自动生成概要设计文档，文档内容参考“软件工程”中相关通用描述


#### 模块2：代码生成Agent（必做）

基于“概要设计Agent”的输出文档“概要设计文档”，生成可运行的源代码：

- 支持至少一种编程语言（Python/Java/TypeScript/C#任选）
- 代码语法正确，可编译通过
- 命名规范、结构清晰
- 包含必要的错误处理


#### 模块3：单元测试Agent（必做）

基于“代码生成Agent”的输出代码和“概要设计文档”，自动生成单元测试： 

- 测试覆盖率目标≥80%
- 测试用例有效、断言清晰
- 可重复执行


#### 模块4：用户交互界面（选做）

提供命令行或Web界面，支持：

- 查看批次执行进度
- 查看各节点状态
- 查看执行日志
- 触发节点执行（手动模式下）


#### 模块5：质量检查机制（选做）

对每个节点的产出物进行自动质量检查，实现自动化质量评分


### 四、技术要求

#### 4.1 基础要求

- 实现语言不限（Python、Java、TypeScript、C#等均可）
- 参赛队伍需提供完整的项目源代码
- 需提供清晰的README文档，包含项目说明、安装步骤、使用指南

#### 4.2 架构要求

- 采用分治（Divide-and-Conquer）的多智能体架构：每个开发环节应由独立定义的智能体代理负责，通过上下文共享与任务拆解实现协同
- 主进程（Orchestrator）负责任务调度和状态管理
- Agent间通信机制：参赛队需自行设计代理间的通讯机制与数据交换格式，以确保上下文在多 Agent 间的有效传递。
- 支持**增量执行**：失败节点可单独重试，无需重新执行整个流程

#### 4.3 数据存储要求

参考以下目录结构组织项目文件：

```
项目根目录/
├── docs/
│   ├── 待生成/                    # 存放输入的产品规格说明书
│   │   └── 采购订单系统规格说明书.md
│   └── 已生成/                    # 存放各节点的产出物
│       └── {批次ID}/
│           ├── batch_status.json   # 批次状态文件
│           ├── execution_log.json  # 执行日志
│           ├── 概要设计/│           
│           ├── 代码生成/
│           └── 单元测试/
├── src/                           # 生成的源代码
├── tests/                         # 生成的测试代码
└── README.md
```

#### 4.4 状态管理要求

使用JSON格式文件管理批次和各节点状态，包含：

- **批次状态**：batch_id、产品规格、状态、当前节点、各节点状态汇总
- **节点状态**：node_id、状态、开始/结束时间、质量检查结果、输入/输出文件列表

### 六、评分标准

| 评分维度 | 权重 | 说明 |
|---------|------|------|
| **功能完整性** | 20% | 必做模块是否全部实现，功能是否可用 |
| **系统架构设计** | 20% | 架构是否合理，模块划分是否清晰，是否采用Subagent机制 |
| **代码质量** | 15% | 代码结构、命名规范、错误处理、注释质量 |
| **流程自动化程度** | 15% | 从输入到产出的自动化程度，节点依赖管理是否完善 |
| **文档与可运行性** | 5% | README是否完整，能否快速搭建和运行 |
| **扩展性与创新性** | 5% | 选做模块实现情况，是否有额外创新功能 |
| **系统验证** | 20% | 实际演示效果、代码可运行程度，系统运行产出物要符合“**4.3 数据存储要求**”|


#### 扣分项

- 缺少README文档或文档不完整：**扣5分**
- 代码无法运行或存在严重语法错误：**扣10分**
- 未采用Subagent机制：**扣10分**
- 提交内容与赛题要求不符：**扣15分**
- Agent 上下文限制：**扣10分**
    Agent 之间传递的数据必须是结构化的文档（Markdown/JSON），严禁将代码直接塞进 Prompt 完成所有任务。
